无线渗透工具

发表于 2018-02-18 更新于 2022-03-01 分类于 无线安全

无线渗透实操

01-aircrack-ng简介

  • 无线渗透和审计神器
  • 包含各种功能的工具套件（而非单个工具）
    • 网络监测
    • 嗅探抓包
    • 包注入
    • 密码破解

  • aircrack-ng仅仅是套件中一个工具，以其来此套件

  • 建议关闭network-maneger服务，存在冲突问题，可能会导致后续操作出错

    service network-maneger stop

  • ifconfig不显示wlan0

    • ifconfig -a是显示所有的，包括未启动的
    • iwconfig会显示（即使没有启动
    • ifconfig wlan0 up启动网卡

02-airmon-ng（侦听）

2.1-检查网卡驱动

  • airmon-ng检查网卡驱动信息，如

    • 显示出当前的驱动信息、目前网卡使用的芯片
    • 驱动问题往往是很棘手的；驱动更新、打补丁

  • airmon-ng check

    • 不是检测网卡的驱动等硬件信息，而是检测aircrack-ng套件在使用过程中与当前系统是否产生冲突的问题

    • 可以手动kill + pid 一个一个删除，但是效率低

    • airmon-ng check kill会把所有可能产生问题的进程全部kill

2.2-开启和停止无线侦听

  • airmon-ng start wlan0

    • iw dev wlan2 interface add wlan2mon type monitor方法也可以，在这利用aircrack-ng中的airmon-ng来实现而已

    • 在wlan0网卡上启动无线侦听，即monitor模式，后续大部分操作都是基于此模式下的，而非默认的managed模式

    • mac80211便是当前的无线协议栈（早期的ieee 80211

    • wlan0变为wlan0mon后enable，并置为monitor模式

    • 同时，disable掉wlan0

    • 再ifconfig后，则出现wlan0mon（wlan0没有了

    • iwconfig后，发现wlan0mon已经是monitor模式

    • 原来wlan0是管理模式，即Mode:Managed，默认模式，此模式下可以正常连接无线网络上网）

  • iwlist wlan0mon channel查看信道信息

    • 支持13个信道
    • 当前工作信道10上

  • airmon-ng start wlan0 3

    • 手动指定其信道
    • 信道号1-13

  • airmon-ng stop wlan0mon关闭监听模式

    • 监听工作结束后，还原至管理模式（原来wlan0是管理模式，即Mode:Managed，默认模式，此模式下可以正常连接无线网络上网）

    • wlan0mon被wlan0取而代之

    • 模式由monitor变为managed

    • 此时ifconfig不可见wlan0（-a可见），因为其未被激活，ifconfig wlan0 up来启动

03-airodump-ng（抓包）

3.1-无线抓包

  • 简介

    • 也可以用wireshark和tcpdump等工具来抓包，但是含大量的其它的数据包（网络层的、传输层的），此外抓到的无线包字段等信息混乱，观察不方便
    • airodump-ng是专门针对无线数据包抓取的，且显示友好（自动对各字段进行排序等操作，便于观察

  • airodump-ng wlan2mon（抓包之前先启用monitor模式）

    • 尽管启动网卡的监听模式后，查看其信道后，其是工作在特定信道上的（可以手工指定）
      • 但是在抓包时，其会处于一种hooping模式，轮循各个信道（1-13），从而得到各个信道中存在的AP
      • 运行后，是动态变化的，空格可暂停其变化

  • airodump-ng wlan0mon -c 3

    • 抓包时指定信道（否则轮循1-13，各个信道都检测
    • 此时结果中会出现PXQ一列（不指定的话便没有）
    • 但是结果中，AP部分也会出现其它信道的AP，因为各个信道是存在交叉重叠的

  • airodump-ng wlan0mon --bssid AE:D1:B8:C7:D4:3F -c 11 -w test

    • -c指定信道

    • –bssid，指定抓取某AP的数据包

    • -w，将结果保存至本地，cap文件可wireshark打开，详细查看其通信

    • 指定特定信道和bssid后，从结果中可以看到连接到它的STA的信息，如MAC地址

  • 其它参数

    • --ivs：Save only captured IVs，适合为破解WEP收集IVs（初始向量），再用这些IVs来破解密码

3.2-输出结果分析

（1）最上面一行：

  • 依次是当前信道（轮循的）
  • 已经工作的时间
  • 系统当前时间
  • 除上述三个字段外，后面还会有其它字段（默认是没有显示的
    • [ paused output提示暂停输出（可按空格停止其动态变化）
    • [ rusumed ouput表正正在输出
    • 其它情况，如果存在WAP/WPA2的AP，有一个客户端连接上AP，其四步握手过程被抓到时，就会显示有一个四步握手被抓到，并将相应的AP的MAC地址显示在此位置（这个数据会被抓取下来）
    • 如果存在一个WEP的AP，有新客户端连接时，会抓到challenge信息（明文的cha和经过密钥流加密的密文cha，二者异或，可以得到密钥流，此时，这个密钥流文件会自动保存到本地，以.xor结尾的一个文件，其文件内容时密文，H可以利用其与AP认证，从而进一步与其建立关联关系，进一步实施攻击，后续WEP攻击会讲到）

（2）上面部分：AP信息

  • BSSID：AP的MAC地址，如win10开的热点

  • PWR：即power

    • 网卡接收到的信号强度，距离越近信号越强
    • 越接近0表示信号强度越强（即绝对值越大越强
      • 小于50：优
      • 小于75：良
      • 大于75：信号较弱
      • 超过130即认为是不可接收，就会显示-1，
    • -1的情况 ：
      • 驱动不支持信号强度
      • 本机距离AP太远，即信号太弱（对本机而言；AP部分的PWR）
      • 本机连接AP，AP上还连接了另一个STA，本机与这个STA相距远（STA部分的PWR）

  • RXQ：最近10秒成功接收的数据帧的百分比（数据帧、管理帧）

    • 数据帧和管理帧中都会有数据内容（不严格来讲，数据帧就是包括数据帧和管理帧的），而控制帧是没有的
    • 是根据接收到的数据帧中的sequence number来判断的，
    • 接受成功的/总接受的，是一个比值（数据是可能接收失败的）
    • 只有在固定信道时才会出现

  • Beacons：接收到此AP发送的beacon帧数量

  • #data：抓到的数据帧数量

    • 可以应用到WEP场景：若是WEP，则此字段可以反映IV数量（WEP要收集 IV来破解密码）
    • 是一个累积的值（而非10秒内等某个时间端的），并且包含广播数据帧（不止单播帧

  • #/s

    • 最近10秒内，每秒平均抓到的帧的数量
    • 不止数据帧，还有其它另外两种，都包含在内

  • CH：信道号（从beacon帧中获得）

    • 信道重叠时可能发现其它信道
    • 即使指定信道，因为信道叠加的原因，也可以发现结果中有其它信道

  • MB：AP支持的最大速率

  • ENC：采用的无线安全技术，如：WEP、WPA、WPA2、OPEN

  • cipher：采用的加密套件，如：CCMP（目前来讲最安全）、TKIP、WEP40、WEP104

  • auth：身份认证方法，如：

    • MGT（WPA企业时，在后端有身份认证服务器，需要账号密码登录 ）、
    • PSK（家用路由器，WPA的预设共享密钥）、
    • SKA（如果WEP而非WPA ，则就是SKA，其便是WEP的预设共享密钥）、
    • OPEN（不要密码的那种）

  • ESSID：无线网络名称

    • 若是隐藏AP，此值可能为空
    • 可以显示隐藏AP的长度，如上图的<length:32>
    • airodump从probe和association request帧中发现隐藏AP
      • AP会隐藏自己，但是要连接他的STA不会帮他隐藏
      • 若airodump-ng抓到相关包，也会在此字段显示出其ESSID

（3）下面部分：STA信息

  • 两种情况
    • STA离本机比较近，故本机网卡也能探测到其的存在
    • STA离本机远（PWR为-1），但是本机网卡与其连接的AP距离近，从而也能探测到这种STA的存在
  • BSSID
    • 若STA关联到AP，则显示AP的MAC地址
    • 若未关联，则（not associated）
  • STATION：终端设备的MAC地址
  • PWR：信号强度，与AP部分的类似，若是-1，则表示本机与其相距太远，攻击就会受影响
  • Rate：传输速率
  • Lost：通过sequence number 判断最近10秒sta发送丢失的数据包数量
    • 包括管理帧、数据帧（二者都含有数据部分，而控制帧没有）
    • 导致Lost太多的原因
      • 干扰、距离
      • 无线网卡使发包不能收、收包不能发的（若处于侦听状态，在收包，其发包就会受影响
  • Frames：STA发送的数据包数量
  • Probe：STA探测的ESSID

3.3-常见问题及解决

  • 不显示任何AP和STA信息
    • 物理机场景下使用笔记本内置无线网卡时，确保BIOS中已启用无线网卡
    • 确认无线网卡在managed模式下可以正常工作
    • 尝试禁用network-managed服务
    • 尝试卸载rmmod和重新加载modprobe驱动
  • 工作一段时间后airodump-ng
    • airmon-ng check kill
    • 确认wpa_supplicant进程已停止

04-aireplay-ng（注入）

4.1-简介

  • 注入包的目的：产生或者加速无线通信流量

  • 向无线网络中注入数据包，可以：

    • 伪造身份验证
    • 强制重新身份验证（伪造正常STA，断掉其与AP的连接，STA会尝试重新连接，再抓包，从而得到相关信息）
    • 抓包，处理后重放（作为中间人）

  • 用于后续WEP和WPA密码破解

  • 格式：aireplay-ng <option> <interface name>（接口需处于monitor模式）

  • aireplay-ng：支持10种包注入的方式（0-9）

  • 获取包的两种途径，即数据包的来源

    • 指定接口抓到的数据包（-i
    • 指定抓包文件pcap（-r

  • Filter 命令选项（除0、1两种攻击方式

    • 抓包后，一般要经过筛选等处理后，再重新发送出去

  • Replay命令选项（重放）

4.2-常见问题及解决

  • aireplay-ng命令挂起没有任何输出
    • 无线网卡与ap工作在不同信道
  • 报错“write failed: Cannot allocate memory wi_write(): illegal seek”
    • 无线网卡使用Broadcom芯片（bcm43xx），替换为b43驱动可解决
  • 可注入但速度很慢，并提示内核消息“rtc: lost some interrupts at 1024Hz”
    • 没有修正方法，此时可以启动多个aireplay-ng命令提高速度
  • 使用-h参数指定注入MAC地址与网卡MAC地址不一致报错
    • 建议保持一致
      • macchanger命令更改MAC地址
      • 更改之前先将网卡down掉
    • 00:11:11:11:11:11 / -h 00:22:22:22:22:22

4.3-aireplay-ng包注入测试

  • 简介

    • 检测网卡是否可以注入包（-9参数）
    • 检测AP的响应时间（回包率反映链路质量）
    • 如果有两个无线网卡，可以检测具体可以注入哪种攻击
      • 只有一个，则只可以-9来最基本的检测
      • 有两个，则可以互相测试，进一步检测可以支持的攻击
    • 基本测试检测AP对probe广播的响应
      • 向AP发30个数据包
      • 网卡成功发送并可接收包的能力

  • 基本测试：aireplay-ng -9 wlan2mon

    • 09:20:16 Injection is working!：表示当前网卡及驱动是支持注入的
    • Trying broadcast probe requests...：发广播包，探测周围存在的AP
    • Found 7 APs：发现了7个AP
    • Trying directed probe requests...：接下来对每个AP进行probe探测
    • 19/30: 63%：每次发30个包，根据其百分比看出其能力（发包及接收包的能力

  • 向隐藏AP/指定ssid注入：aireplay-ng -9 -e huazhu-hanting -a 88:25:93:CA:98:A2 wlan0mon

    • -e：指定ESSID，即无线名字
    • -a：指定无线AP的MAC地址
    • 此时就不是一个广播了，是由针对性的单播
    • 若信道不一致，则更换一个与wlan0mon同信道的AP来探测（aireplay-ng无法手动指定信道，只能换AP来测试）

  • card to card注入测试

    • 若有两块无线网卡，都置于monitor模式，可以做网卡-网卡的注入
    • 检测结果更加准确、精细，可以检测0-8这几种方式是否都支持
    • 网卡a去发包，发到b，网卡b接受包（b模拟AP接受数据包），若b能接受并返回包，则说明二者支持这种类型的包
    • -i参数：指定作为AP的网卡（另外一个网卡发包用
    • 如图：5/7 Failed
      • 可能是真的不支持这两种类型（但是在其它类型都支持的情况下，这种可能性很小
      • 实践中：只要保证注入Mac和真实Mac相同，那么是可以正常使用的（-h指定
