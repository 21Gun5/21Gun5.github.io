果加智能门锁的安全分析

上

  • 概要：抓包获取固件、MCU手册、固件解析（指令集、加载基址、解析位置）
  • 设备-中转-云端的通信方式
    • 设备-BLE协议-手机app-云端（手机无线通信，无需交换机）
    • 设备-433MHz无线-设备网关-交换机-云端（设备网关一般是有线，故有交换机）
    • 留意其它方式
  • 固件
    • 传统意义：驱动硬件的软件程序
    • IoT设备中：包含了RTOS(实时操作系统)和应用软件，提供了从底层驱动（操作系统）到高层应用（应用软件）的所有功能
    • 固件一般要包含一个RTOS，设备上电后就会启动RTOS（比如Linux系统就启动kernel内核）
    • 目前遇到两种情况：Linux系统则Linux kernel+文件系统中程序、Vxwork等系统则一个单一可执行文件
  • 固件获取
    • 手机中转：设备-手机app-云端，设备更新一定会访问云端，而手机app作为中转，故fiddler抓包手机
    • 网关中转：设备-网关-交换机-云端，对交换机做端口监控（步骤未知//here）
  • fiddler抓包手机
    • fiddler设置两处：解密https通信（固件更新可能会https）、允许其它设备连接代理（而非只有运行fiddler的pc）
    • 手机设置两处：设置根证书（解密https用）、将fiddler作为代理转发数据（即设备-手机-fiddler-云端）
  • 逆向对象
    • Linux：包含Linux文件系统，文件系统中的各个程序将是分析重点
    • FreeRTOS/Vxworks：固件是一段编译好的包含代码和数据的可执行文件，单一的单片机程序
  • 自动/人工识别
    • 若是Linux文件系统中的程序，可自动识别：ELF等格式、处理器型号
    • 若固件是单一的可执行程序：与设备硬件相关，不同MCU会使用不同的指令集、文件系统等，需人工识别
  • binwalk
    • 情况1：识别出U-Boot引导程序、Squashfs文件系统、lzma等压缩数据、其它（最普遍的情况，直接-Me提取）
    • 情况2：无法辨别固件，即没有任何输出，固件可能：可执行文件（需直接逆向固件）、加密过的（先解密再提取）
  • 指令集
    • 型号：根据MCU上刻印的芯片型号-STM32L071，找到对应的手册
    • datasheet：目录introductory，使用了Arm Cortex-M0+内核
    • www.arm.com ：查询该内核采用Armv6-M架构，即Armv6-M指令集
    • 设置IDA：processor options、edit arm architecture option、选择Armv6-M
  • 加载基址
    • 固件程序没有重定位表等结构，若加载地址错误，就会出现不能解析的跳转和全局变量
    • 通常MCU都会有一块固定的存储区域用于存储代码，这个区域的起始地址就是固件的加载地址
    • References Manual：目录Memory organization，找内存映射图，固件位于CODE的Flash System Memory区域
    • 设置IDA：ROM start address、Loading address
  • 解析位置
    • 指定指令集、加载地址还不够，还需指定解析位置
    • programming manual：目录vector table，找中断向量表：位置（固件起始）、结构（第二项是reset）
    • 设备上电之后执行的代码，即Reset复位中断（reset事件对应reset函数）
    • reset位于第二项即偏移0x4，跳转到此，按’c’将此处解析为代码
  • 手册
    • Datasheet即产品规格：目录memory mapping，找内存映射图（提示到reference manual ）、目录introductory，找采取的内核
    • References Manual即参考手册：目录Memory organization，找内存映射图
    • Programming Mannual即编程手册：目录vector table，找中断向量表的位置及结构
  • 芯片-内核-架构-指令集
    • 芯片采用Arm Cortex-M0+内核
    • Arm Cortex-M0+内核采用Armv6-M架构
    • Armv6-M架构即对应Armv6-M指令集

中

  • 概要：通过app中常量定位固件中代码、门锁固件分析

  • 固件分析

    • 分析难度大：代码多、没有调试信息、没有可参考的字符串
    • 以BLE通信为突破口，找到关键代码
    • 法1：通过BLE通信的帧格式：在代码中定位格式中常量，通过常量找到BLE通信数据的解析函数
    • 法2：通过硬件电路：找到MCU与BLE芯片的通信引脚，查看文档确定该引脚映射的内存地址，找到BLE发送/接收函数

  • app分析：获取常量

    • 思路：门锁与app进行通信，先分析app找到常量，再利用常量去固件中定位
    • java层：通过日志字符串，定位到libBleCmd.so库文件
    • Native层：zip解压apk获取so文件、导出函数中发现Tea算法相关的加/解密函数、Tea算法会用到常量、找到2个常量0xCE6D-0x58BF

  • 固件分析：通过常量定位

    • 设置架构为Armv6-M，有些指令无法反汇编，改用Armv7-M后成功，原因未知。
    • alt + t搜索常量，再对比libBleCmd.so-Decrypt，确定固件中Encrypt/Decrypt函数

  • sub_800B528调用Decrypt

    • 0xBABEC0DE常量：app日志中，0xBABEC0DE为BLE消息开头
    • Decrypt的3参数：密文、长度、解密密钥
    • 解密密钥：动态生成的，放于变量中（内存中），进一步找到密钥生成代码
    • 小结：ble消息预处理，即消息头检验、crc校验、是否可解密
  • 代码进一步分析
    • sub_800EB20调用sub_800B528：根据其返回值设置内存（即变量赋值）、密文及长度的内存
    • sub_80000C8调用sub_800EB20：见下篇
  • 小结：
    • app分析：日志分析、先定位到java再native、so中找到加解密函数、tea算法中常量
    • 门锁固件分析：常量定位到加解密函数、分析出密文-长度-密钥的内存、不断交叉引用加强程序理解
  • 其它
    • Armv6-M是Armv7-M的子集，v6程序可直接移植到v7-M架构上
    • 反编译工具，JEB相比于Jadx，更方便
    • java/native层：Java程序中，可以使用由其他编程语言实现的函数，即原生（Native）函数

下

  • 概要：固件代码分析、重刷固件、语音提示函数、测试模式、后门密码

  • sub_80000C8即main

    • 最复杂的函数，梳理代码框架，是个while死循环，只有入口没有出口
    • 系统复位中断之后执行的第二个函数
    • 综上：最复杂+死循环+第二个 = main

  • 相关概念

    • 复位中断会执行两个函数：init、main
    • 单片机：main一般是死循环，不可执行完毕后退出。上电之后，MCU就开始运行程序，直到断电
    • 运行于操作系统之上的程序：如ls，完成功能后就退出

  • main

    • 一般流程（个人经验）：消息预处理、BLE指令解析、完成逻辑功能
    • 调用sub_800EB20：返回0表预处理成功，即消息头校验、crc校验、数据解密
    • 两个标志位：1-是否收到ble指令
    • 调用sub_800F9EC：switch-case语句处理各种ble指令、结合日志推断变量含义

  • case 0xE004

    • ble指令类型：开锁，即门锁如何处理开锁指令
    • sub_80125C8：日期转时间戳，不在区间时调用sub_800D40C（暂未知功能），即密码过期不可开锁

  • sub_800D40C语音提示函数

    • 调用位置很多、参数为常量、main进入while前调用
    • 反复修改并刷入固件（main-while那个），发现提示音不同
    • 小结：语音提示函数、参数即语音内容、类似日志输出可帮助理解代码、整理出语音表

  • 固件重刷

    • 修改：16进制编辑器
    • 刷入：固件更新时作劫持，Fiddler-AutoResponder，新固件替换原固件

  • 语音“进入测试模式”

    • 如何触发语音，即如何进入测试模式？
    • 判断某函数返回值，其参数为字符串（即门锁键盘可输入的值），应该类似strcmp
    • 输入此字符串，门锁果然提示说“进入测试模式”

  • 后门密码：测试模式下，2个数字加#号，可开锁

符号执行的基础知识

发表于 2021-05-21 更新于 2022-02-21 分类于 软件安全/逆向

符号执行的基础知识

  • 概要：符号执行、KLEE、LLVM、Inception

  • 符号执行（symbolic execution）

    • Wiki：通过分析程序，确定哪些输入向量，会对应导致程序的执行结果向量

    • 通俗：给定英雄的最终属性值，分析出通过哪些出装，达到这种效果

    • 白盒：分析程序输入，需要获取到目标源代码

    • 静态：并没有实际执行程序，而是分析其执行路径

    • 挖洞：英雄的最终属性值-程序bug状态，可用来挖掘漏洞的输入向量

  • 代码实例

    • 输入M,N,Q，输出x1,x2,x3和，什么样的输入向量<M,N,Q>，得到的三个输出变量和等于3？
    • 树形结构列出所有情况，叶子节点即输出，当~M^(N<5)^Q时，结果等于3
    • 可逆向发现输入向量，如果把结果改为漏洞条件，可漏洞挖掘

  • 根据结果求输入

    • 约束规划的求解问题
    • 数学工具，如SMT（Satisfiability Modulo Theory，可满足性模理论）和SAT

  • 过程截断问题

    • 实际的程序更复杂，包含与外设交互的系统函数
    • 与外设交互的系统函数，其输入输出不会直接赋值到符号，故过程阻断
    • 如示例中，文件读写操作，fputs写入文件后，fgets再读取文件，读取依赖前面的写入，因为写入了文件，故阻断

  • 解决过程截断

    • 经典项目KLEE（base LLVM）：重写与外设有关的系统函数，使符号数值的传递继续下去
    • 思路简要：修改函数fputs写入变量、fgets读取变量（而非文件，变量的值可以传递）
    • KLEE中，重新对40个系统调用进行了建模，如open, read, write, stat, lseek, ftruncate, ioctl

  • 嵌入式设备的漏洞挖掘

    • 符号执行技术挖掘漏洞依然可行
    • 过程截断问题：类似KLEE中与外设交互的系统调用建模
    • 混合代码问题：嵌入式源代码中有c语言/二进制/汇编代码（写汇编的是为了提高程序执行效率）

  • 通过LLVM转换成LLVM-IR

    • LLVM-IR：LLVM中间的语言表达形式，也是一种汇编语言
    • KLEE就是LLVM-IR工具实现的符号执行虚拟机

  • LLVM-IR的变量有三种，通过前缀@或者%区分，@表示全局变量，%表示局部变量

    • %或者@接数字，表示的是临时变量，在一个函数中，从0开始编号使用。比如%0，%1
    • %或者@接字符串，表示有名字的变量，可以任意使用；
    • 第三类就是立即数

  • LLVM-IR再次转化

    • 转换成LLVM-IR后，由于代码含有arm汇编，而其中没有调用参数的代码和方式，使得符号数值的传递再次中断，导致KLEE不能执行

    • 需要对混合代码再次转化，使得arm汇编也能够被KLEE的符号执行虚拟机分析

    • USENIX Security18论文Inception，将混有高级/低级语言的代码同时转化成KLEE能够分析的语言

  • 小结

    • 现有的符号执行工具，主要基于KLEE，深入理解KLEE，有助于打造更加高效的工具
    • Inception工具针对ARM架构，而MIPS尚未有类似的符号执行工具
    • 基于IDA的脚本工具bugscam，修改后能够支持MIPS，然而误报率非常高
    • 如何符号执行 + IDA = 具有符号执行的MIPS漏洞分析工具 ？

云丁鹿客智能门锁的安全分析

上

  • 概述：app与门锁的ble通信、app端（下篇是门锁固件端）
  • 如何分析
    • 切入点：门锁可通过手机BLE开锁，故从app的开锁流程开始
    • 方法：日志、流量、逆向
  • Challenge/Response模式
    • a：通知指令：app与门锁建立BLE连接后，当使用app开启门锁时，app会先下发一条通知指令，告知门锁准备进入开锁流程
    • b：Challenge：门锁向手机发送一组数字，称为Challenge；
    • c：Response：app对Challenge进行处理生成Response并发送给门锁，门锁验证Response正确时，就会开锁
    • 何处认证：多次开锁，a仅一个字节（序列号）不同，故a握手作用，认证在b和c
  • 思路
    • 门锁返回challenge、aes-ebc加密、异或固定数据
    • ebc加密密钥BleKey：reset_token请求服务器、aes-cbc解密返回包中token-token即
    • cbc解密密钥CryptSecret：crypt_cecret请求服务器、返回包中secret即
  • 推测
    • a：每次app与门锁建立通信时，都会请求一个Bletoken-BleKey并下发给门锁
    • b：当app与门锁建立通信时，如果没有Bletoken-BleKey，才会向服务器发送请求
    • 证伪a：手机重复开启门锁，日志和流量中都没有http请求，故本地保存了，不需下载
    • 证明b：清空app缓存，删除本地存储的BleKey，再开锁，出现http请求
  • app通信
    • app-服务器：aes-cbc解密 + reset_token返回包中token-token = ebc加密密钥
    • app-服务器：crypt_cecret返回包中secret = cbc解密密钥
    • app-服务器：base64解码 + reset_token返回包中buf-totalData = 门锁解密的BLeKey
    • app-门锁：sendBleKeyCommandNewProtocol，base64+totalData，下发BleKey，ble通信可被监听）
    • app-门锁：sendUnlockCommandNewProtocol，处理challenge后生成的response，开锁指令
  • 门锁校验
    • app：发送通知，即将开锁
    • 门锁：发送challenge
    • app：发送challenge加密后的response、加密密钥
    • 门锁：密钥解密response，对比明文与原始的challenge
  • 开锁流程图
    • 本地没有BleKey：app请求CryptSecret（密钥）和BleToken（密文）、aes-cbc解密BleToken-token、明文即BleKey
    • 本地有BleKey：aes-ebc加密challenge（BleKey为密钥）、密文异或固定数据=response、门锁对Response校验

中

  • 概述：固件保存在MCU内置flash存储器中、通过MCU的调试接口提取固件
  • 软件调试：
    • 依赖Linux系统提供的软件调试接口
    • gdbserver软件调试器（ida/gdb作为client）
  • 硬件调试：
    • 依赖MCU提供的硬件调试接口，SWD/JTAG
    • 需要相对应的硬件调试器，如JLink（支持SWD/JATG）、STLink（支持SWD）
  • MCU芯片手册，可找到
    • mcu的核心：ARM Cortex-M3，ida加载固件时会设置
    • MCU提供的调试方式：何种调试接口，SWD/JTAG
    • 调试接口的引脚定义：如何将mcu与硬件调试器相连
    • 代码在内存中的位置：代码即狭义上固件、savebin提取固件会用到位置
  • 硬件调试器：
    • 连接PC与MCU，一端的USB口连PC，另一端连MCU
    • 连接MCU的接口有多种，如SWD/JTAG
    • 不同的调试接口，引脚定义也不同
    • 通常会提供相应的软件工具，比如JLink Commander命令行工具
  • 读保护
    • 原则上，通过硬件调试器连接MCU提供的硬件调试接口，可读取MCU内置flash中的数据，即获取固件
    • 芯片可开启读保护机制（Readout Protection）防止读取
  • 连线
    • JLink的引脚定义，说的就是连mcu的哪一个，如Vtref即电源，就连接MCU的3.3V接口
    • 若接口不好连，可提前焊接排针
  • 固件提取
    • 状态检测，显示基本信息（信息可能不对，假货）
    • connect：连接MCU接口
    • ?指令，弹框，选择芯片的厂商及型号
    • s选择调试接口类型为SWD（默认JTAG）
    • 选择通信速率，默认即可
    • 显示内存信息，准备工作完成
    • savebin + 固件名 + 开始地址 + 结束地址
  • 固件解析
    • 同属Cortex-M系列，所以分析方法就相同，核心是啥？//here
    • 固件基址：代码在内存中的起始地址
    • 寻找Reset中断的中断服务程序，跳转之后按“c”键（原理看其它文章//here）
  • 固件到底是啥？
    • binwalk+固件：得到文件系统
    • vxworks固件：就是一个大文件
    • 本文固件：是内存中的代码
  • 各种手册
    • reference-manuals，参考手册，目录6 DBG-Debug Interface，确认是SWD调试、调试引脚（参考手册有多份，在EFM32GG这份）
    • data-sheets，数据表/产品规格，目录1. Feature List，确定核心为：ARM Cortex-M3
    • data-sheets，数据表/产品规格目录3.3 Memory Map，确定flash（即代码即固件）在内存中的位置为0-0x100000，固件基址即0

    • JLink：不仅可提取固件，也可动态调试
    • Ozone：SEGGER提供的图形界面工具，可展示多个subview
    • 创建新项目：选择芯片型号、调试接口类型、通信速率等

    • 门锁与app的BLE通信，有AES加密保护，故从AES入手
    • 芯片手册：提供AES处理模块，位于0x400E0000~0x400E0400
    • 假设使用芯片的AES模块，而非自写的AES算法，那么必然访问这片内存
    • 设置内存读写断点，手机上点击开锁，找到AES的处理函数，即AESFunc

    • body：多个数据块，其结构：1类型-2长度-len内容
  • AESEntry函数
    • AESFunc的封装，totalData生成BleKey的核心代码
    • 流程：totaldata、aes解密（需要根密钥）、异或、blekey
